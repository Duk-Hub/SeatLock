# 01. race condition 재현 - 락 도입 전 기록

> 좌석 예매에 동시성 제어가 없는 상태(#16 시점)에서, 같은 좌석을 동시에 예매할 때 무슨 일이 벌어지는지 재현·측정한 기록.
> `ReservationConcurrencyTest`는 정합성 방어가 없는 상태를 assert로 실증하며, 락 도입 시 검증 조건이 정합성 보장으로 바뀐다.

## 실험 설계

- Testcontainers MySQL 8.4 (InnoDB, 기본 격리수준 REPEATABLE READ), HikariCP 풀 10 (기본값)
- 같은 좌석 1개에 32개 스레드가 `CountDownLatch`로 동시 출발, `ReservationService.reserve()` 직접 호출
- 실험① 현재 스키마 그대로 / 실험② FK 제약을 잠시 제거하고 동일 실험 (테스트 종료 시 복원)

재실행: `./gradlew test --tests '*ReservationConcurrencyTest'` (Docker 필요)

## 결과

| 실험 | 성공 | 409 (정상 거절) | 데드락 (사용자에겐 500) | 좌석 1개의 예약 행 |
|---|---|---|---|---|
| ① 현재 스키마 (1차 실행) | 1 | 22 | 9 | 1 |
| ① 현재 스키마 (2차 실행) | **2** | 22 | 8 | **2 ← 오버부킹** |
| ② FK 제거 실험 | **10** | 22 | 0 | **10 ← 오버부킹** |

실험②의 로그 — 같은 밀리초에 10개 스레드가 좌석 1번을 읽고(.007), 같은 밀리초에 전원 "예매 완료"(.086):

```
12:52:54.007 [pool-2-thread-2]  예매 시도: memberId=1 scheduleSeatIds=[1]
12:52:54.007 [pool-2-thread-16] 예매 시도: memberId=1 scheduleSeatIds=[1]
   ... (동시 시도 총 10건) ...
12:52:54.086 [pool-2-thread-7]  예매 완료: reservationId=7  memberId=1 scheduleSeatIds=[1]
12:52:54.086 [pool-2-thread-26] 예매 완료: reservationId=5  memberId=1 scheduleSeatIds=[1]
12:52:54.086 [pool-2-thread-19] 예매 완료: reservationId=4  memberId=1 scheduleSeatIds=[1]
   ... (완료 총 10건, reservationId 1~10이 전부 같은 좌석(scheduleSeatIds=[1])) ...
```

## 해석

**1. 코드에는 정합성 방어가 없다.**
예매 로직은 조회(락 없는 스냅샷 읽기) 후 상태를 저장하는 check-then-act 구조다. 동시 스레드 전원이 같은 좌석을 AVAILABLE로 읽고 전원 상태 가드를 통과한다. 실험②에서 커넥션 풀을 잡은 첫 웨이브 10개 스레드가 전원 성공했다 — 좌석 1개에 유효한 예약 10건.

**2. 실험①에서 오버부킹 대신 데드락이 나오는 이유(실험②의 필요성)**
`reservation_seat` INSERT의 FK 존재 검사가 부모 행(schedule_seat)에 공유 락을 걸고, 직후 UPDATE가 같은 행에 배타 락을 요구한다. 동시 트랜잭션들이 서로의 공유 락에 막혀 순환 대기에 빠지고, InnoDB가 하나만 남을 때까지 강제 롤백한다. 즉 실험①에서 중복 예매를 막은 주체는 애플리케이션 코드가 아니라 FK 검사의 부수효과다 — 이를 걷어내고 코드만의 방어력을 확인한 것이 실험②다.

**3. 결론: 명시적 동시성 제어가 필요하다.**
FK의 부수효과는 신뢰할 수 없고(2차 실행에서는 FK가 있어도 오버부킹 2건), 그 비용으로 동시 진입자의 80~90%가 "이미 예매된 좌석입니다"(409)가 아닌 데드락 예외(500)를 받는다. 정합성도 사용자 경험도 지켜지지 않는 상태다.

## 다음 단계 (락 도입 시 검증 조건 반전)

- 실험① `데드락 > 0` → `= 0` — 락이 데드락을 제거함을 증명
- 실험② `성공 > 1`, `예약 행 > 1` → `= 1` — FK의 도움 없이 코드가 정합성을 보장함을 증명
